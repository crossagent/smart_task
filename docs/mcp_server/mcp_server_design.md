# 模块设计文档：`mcp_server` (服务总线层)

## 1. 模块定义 (Module Definition)
`mcp_server` 是系统的唯一对口服务门户。它充当了系统与外部环境（如 IDE、AI 智能体、或网关）之间的通信桥梁。它的主要功能是封装底层的业务逻辑，以标准化的 Model Context Protocol (MCP) 协议将功能暴露给外部，并负责整个应用程序生命周期的冷启动引导。

## 2. 模块接口 (Module Interface)
- **输入 (Input)**：
    - 外部发起的标准化工具调用指令（如 `query_sql`, `upsert_task` 等请求）。
    - 运行环境配置信息（通过环境变量指定监听端口、传输模式如 `stdio` 或 `http`）。
- **输出 (Output)**：
    - 结构化的 JSON 响应数据，包含工具执行的结果或错误信息。
    - 服务端的实时日志输出与事件流（Event Stream）。

## 3. 模块流程 (Module Flow)
1. **引导初始化**：服务器启动，首先根据配置确定通信协议（标准输入输出或 HTTP）。
2. **后台唤醒**：在主通信循环开始之前，静默拉起任务调度引擎（Scheduler）的守护线程，使其在后台独立运行。
3. **工具挂载**：从底层业务模块（Task Management）中动态导入并注册所有可用的操作工具。
4. **请求监听**：进入阻塞监听状态，等待外部指令。
5. **分发路由**：接收到指令后，将其映射到对应的业务函数执行。
6. **响应回传**：将业务层的执行结果封装成符合协议格式的响应发送给客户端。

---

## 🚫 修改禁忌与稳定性政策 (Modification Prohibition Policy)

> [!CAUTION]
> **本模块作为系统的静态基础设施层，除非发生以下底层核心变更，否则严禁执行任何代码修改：**
> 1. 需要升级底层 `mcp` 官方库以适配新版本协议特性。
> 2. 需要调整物理网络传输协议（例如从目前的 `stdio` 全域切换到 `http/sse` 的内核参数）。
> 3. 需要增加系统级的拦截器（如全域限流 Rate Limit、全域审计日志或统一身份校验）。

**严禁在此执行以下操作：**
- **严禁在此层定义任何具体业务逻辑或 SQL 查询语句**。
- **严禁在此层硬编码任何特定工具的内容**（必须在 `task_management` 中定义并通过接口挂载）。
- **严禁在此层定义具体的任务流转规则**（那是调度器的职责）。

**核心目的**：确保在多智能体并发并行任务下，作为“总线控制中心”的 `mcp_server` 始终保持绝对的确定性与零崩溃率。
